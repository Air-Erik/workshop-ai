# Кейс 1: Рефакторинг бэкэнда без изменения поведения

## Контекст задачи
- Есть сервис, который вырос и стал трудно читаемым.
- Нужен рефакторинг без изменения логики.

## Плохой промпт
- Слишком общий, без критериев приемки.

## Исправленный промпт
- Цель → контекст → ограничения → критерии → формат ответа.

## Результат
- Четкий план и патч с минимальными правками.

## Выводы
- Чем точнее критерии, тем меньше лишних изменений.
